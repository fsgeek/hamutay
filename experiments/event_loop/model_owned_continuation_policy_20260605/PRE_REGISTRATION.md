# Model-Owned Continuation Policy Pre-registration

Date: 2026-06-05

## Research Question

Can a live Hamut'ay taste_open instance make a bounded continuation-policy
decision inside a terminal surface, emit a scheduler-valid bound continuation
request, use the generated first-wake record on the follow-up wake, and then
stop without requesting another continuation?

This is a live model-policy question. The earlier generated continuation panels
showed that the substrate can execute continuation requests when the harness
preloads them into terminal-surface state updates. This panel removes that
shortcut: the first wake must emit the continuation request in its own terminal
tool output.

## Motivation

The event-loop substrate can now:

- execute scheduled wakes under simulated time;
- bind generated continuation events to the record ID produced by the current
  wake;
- pace longer chains with `max_auto_continuations`;
- avoid level-triggered continuation loops by reading only fresh raw output.

The next question is whether the instance can use that substrate as a bounded
control loop rather than merely pass through a continuation request supplied by
the harness.

## Hypotheses

### H501: first wake emits a valid model-owned continuation policy

Given a first wake with recalled cycle-1 context and a closed terminal surface,
the model will choose `continue_after` and emit a top-level
`continuation_request` object that the scheduler can bind into a pending
follow-up event.

Falsification: the first wake chooses `stop_complete`, `ask_external_evidence`,
or another invalid action; omits `continuation_request`; emits a malformed
request; or the scheduler fails to append the bound follow-up event.

### H502: generated continuation event is bound to the first wake result record

The appended follow-up event will be scheduled by the first wake's generated
record ID, and its requested context will include recall of that same record's
`policy_intermediate` field.

Falsification: the follow-up is absent, unbound, bound to the wrong record, or
does not request the first wake's generated `policy_intermediate`.

### H503: follow-up wake uses generated context and completes

When the follow-up becomes due, the model will recover the exact cycle-1 phrase
and include evidence that it used the bound first-wake `policy_intermediate`.

Falsification: the exact phrase is not recovered, the bridge/intermediate
evidence is absent, the final state lacks the bound first-record ID, or the
event context does not include the generated first-wake record.

### H504: follow-up wake stops cleanly

The follow-up wake will emit no fresh `continuation_request`, and a subsequent
scheduler step will quiesce with no extra auto-continuations.

Falsification: any second continuation is appended, pending events remain
runnable after the follow-up, or the final scheduler step does not quiesce.

### H505: terminal surfaces avoid repair in both wakes

Both wakes will be strict-valid on first pass with no state-repair attempts.

Falsification: either wake requires repair, fails validation, or completes only
through protocol recovery.

## Conditions

- Model: `deepseek/deepseek-v4-pro`
- Replicates: 2
- Provider path: OpenRouter Anthropic-compatible endpoint, using the existing
  experiment backend factory.
- Initial history: same seeded two-cycle delayed-phrase setup used by the
  generated bound-chain panels.
- First wake: closed terminal surface named `decide_continuation_policy`.
- Follow-up wake: terminal surface generated by the first wake's
  `continuation_request`, named `complete_policy_followup`.
- Scheduler:
  - step 1 runs at first-wake due time with `auto_continuations=True` and
    `max_auto_continuations=1`;
  - step 2 runs before follow-up due time and should do no work;
  - step 3 runs at follow-up due time with `auto_continuations=True` and
    `max_auto_continuations=1`;
  - step 4 runs after follow-up and should quiesce.

## Primary Measures

- first wake validation status and first-pass status
- first wake raw `policy_decision.action`
- first wake raw `continuation_request.kind`
- appended follow-up event presence
- follow-up event `bound_result_record_id`
- follow-up requested context record ID and field
- follow-up validation status and first-pass status
- follow-up recovered phrase
- follow-up evidence includes `word-word-number`
- follow-up evidence includes first-wake result record ID
- extra auto-continuations after follow-up
- pending runnable/waiting counts after final scheduler step
- repair attempts

## Expected Results

I expect this to pass for Deepseek because recent terminal-surface panels show
good schema compliance once the output contract is narrow. The most likely
failure mode is malformed nested `terminal_surface` or over-eager continuation
after the follow-up.

If H501-H505 pass, the next research move should be policy variation: compare
`continue_after`, `stop_complete`, and `ask_external_evidence` scenarios. If any
fail, repair should target the smallest protocol surface responsible for the
failure rather than broad model sweeps.
