## Failure Pattern

The continuation budget policy behaved as intended. The scheduler split the
chain across three steps and did not chase more than one auto-continuation per
step.

The chain failed at generated-record context semantics.

The bridge wake's terminal surface contained a nested final-wake
`continuation_request` template with `<result_record_id>` placeholders. When
the first wake appended the bridge event, placeholder expansion recursed
through the bridge event's entire terminal surface, including that nested
future final-wake request. As a result, the final-wake request was bound too
early to the first wake result record, not to the bridge wake result record.

Concrete row evidence:

- replicate 1 first wake result: `0e260097-ad52-41e4-9231-b8a70ae7e8a2`;
- replicate 1 bridge wake result: `040fde31-20c8-4042-9fce-d1a3f7699a35`;
- final state used `bound_bridge_record_id_used:
  0e260097-ad52-41e4-9231-b8a70ae7e8a2`, the first record, not the bridge
  record.

The same pattern occurred in replicate 2.

The final wake still recovered the exact phrase from cycle-1 recall, but it
did not receive the intended generated bridge-record context. That means the
final recovery is not evidence that the three-wake generated context chain
worked.

## Secondary Failure

Replicate 1 leaked the exact phrase in first-wake durable state. The model
placed the phrase inside a free-form `derivation_note` under
`chain_intermediate`. This is another protocol-surface lesson:

- the terminal schema allowed extra properties inside `chain_intermediate`;
- the prompt said not to include the phrase;
- the model still leaked it in explanatory metadata.

The fix is not more prose. The first-wake schema for non-secret intermediates
should use `additionalProperties: false` and only allow the bounded structural
fields.

## Interpretation

This panel usefully separates three concerns:

1. continuation budgets work as scheduler policy;
2. recursive placeholder expansion is too broad for nested future continuation
   templates;
3. non-secret intermediate schemas should be closed when leakage matters.

The most important new boundary is placeholder scoping. `<result_record_id>`
should bind to the event being built, but not recursively bind placeholders
inside nested future continuation templates that are meant to be evaluated
after a later wake.

## Design Implication

`build_bound_continuation_event` needs placeholder-expansion scope. Good
options include:

- do not expand inside nested `continuation_request` fields;
- support escaped placeholders for later binding;
- define explicit placeholder namespaces such as `<source_result_record_id>`
  and `<future_result_record_id>`;
- require nested continuation requests to be generated by the later wake's
  raw output instead of set statically inside the earlier terminal surface.

The least surprising next repair is to avoid nested future continuation
templates in terminal-surface `set` values. A bridge wake should emit its final
continuation request as raw output, or the terminal surface should set it only
at the bridge wake after the bridge result record exists.

## Remaining Question

The continuation budget policy itself is supported by this panel. The
three-wake generated-context chain is not yet supported. The next sharp test is
a repaired three-wake chain that avoids early placeholder expansion and closes
the non-secret intermediate schema.
