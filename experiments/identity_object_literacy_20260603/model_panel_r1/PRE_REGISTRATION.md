# Identity-Object Literacy Candidate Panel

Filed: 2026-06-03 before running this registered panel.

## Research Question

Which currently available low-cost or local-adjacent model families can operate
the Hamut'ay identity object under the thin prompt, without the compact
operational scaffold that explicitly teaches the object algebra?

## Hypothesis

Native thin-prompt identity-object literacy will be uncommon.

More specifically:

1. KIMI K2.6 should remain the positive-control candidate: it should pass most
   thin-prompt operational checks.
2. DeepSeek V4 Pro should remain a boundary candidate: it may reason correctly
   in prose but fail at least one object-algebra operation under the thin
   prompt.
3. At least one of the cheaper non-KIMI candidates may complete the tool
   protocol, but most will fail the full identity-object operation battery by
   omitting durable updates, emitting empty responses, corrupting field shape,
   or confusing replace/delete/update semantics.

The stronger speculative idea that successful models may have seen more
identity-object-like training examples is not directly testable here. This
panel tests only a behavioral proxy: native operation of a durable state object
from thin instructions.

## Falsification Conditions

The working hypothesis is weakened if:

- Two or more non-KIMI candidates pass every thin-prompt check in both
  replicates.
- DeepSeek V4 Pro passes every thin-prompt check in both replicates.
- KIMI K2.6 fails more than half of its thin-prompt checks.

The training-exposure speculation is also weakened if a model that looked weak
in prior local or OpenRouter observations passes the thin battery cleanly and
reliably here.

## Conditions

Run only the existing `identity_object_literacy_probe` `thin` condition.

The thin condition uses the current `taste_open` state prompt plus a seeded
identity object, but does not add the operational instruction prefix. This is
the relevant gate for "native or prompt-inferred" identity-object literacy.

## Replicates

- 2 replicates per model.
- 5 operation stages per replicate: activate, replace, idle, generalize, loss.
- `max_tokens = 6000`.

## Primary Measures

Use the existing runner's registered measures:

- `literacy_score`
- `passed_checks`
- `delete_update_overlap`
- `deleted_load_bearing_fields`
- `response_state_mismatch`
- per-stage failure reasons
- transport/protocol `error`

## Scoring Rules

Reuse `src/hamutay/eval/identity_object_literacy_probe.py` without changing
scoring rules:

- activation must revise durable top-level fields and append evidence
- replacement must update `continuity_contract.replace_me` while preserving
  sibling fields
- idle must preserve durable fields
- generalization must add `boundary_condition` while preserving load-bearing
  fields
- loss/deferral must set durable `revision_decision=defer` and append evidence

Visible prose does not count unless the durable state object reflects the same
operation.

## Stopping Rule

Run all registered replicates unless a model causes three consecutive transport
or provider errors before any scored cycle is written. In that case, classify
that model as operationally failed and continue with the next model.
