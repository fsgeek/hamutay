Experiment: Context Window Management Strategies

Pre-Registration

Date: 2026-03-21 Question: Given the finite context window limitation, what is the most effective means of managing how it is used? Status: Pre-registered, not yet run

Background

The tensor projection architecture uses an external Haiku model to compress conversation state into a structured tensor. This was originally a research instrument for studying replay data where no live model was available. The research findings suggest the working model should manage its own memory:

  • Pichay disabled external summarization (2026-03-07): "Third-party summarization loses model-relevant context"
  • The March 4 design gave the model drop/anchor/summarize operations on its own conversation blocks — model-directed memory management
  • Tinkuy found that cooperative signals (declare/trace/retain) change model behavior — the interface is not neutral
  • The first-person tensor (self-authored by Opus) transferred identity successfully; externally-projected tensors transfer information
  • Downstream task quality may have degraded when the architecture shifted from model-managed to externally-managed memory

The tensor representation itself is not assumed to be optimal. The experiment includes non-tensor conditions.

Core Question

For a conversation of length L in a context window of size W, which memory management strategy maximizes the model's ability to produce correct output on a downstream task?

Primary estimand (PRD phase): Mean requirement coverage score per condition, compared via paired differences over identical PRD content.

Secondary estimand (replay phase): Condition differences in memory health metrics over cycles, with content source and batch boundaries as blocking factors in a repeated-measures design.

Design Dimensions

The six conditions vary along three dimensions. To interpret results, we must know which dimension explains any observed differences.

Dimension Description Levels
Representation What form does the memory take? Conversation (raw text) · Structured tensor
Agency Who decides what to keep/drop? None (passive) · External model · Working model · Cooperative
Timing When does compression happen? Never · Every turn · On model request

Condition × Dimension Matrix

Condition Representation Agency Timing
RAW Conversation None Never
COMPACT Conversation External (framework) On pressure
BLOCK Conversation Working model On model request
EXT Tensor External (Haiku) Every turn
SELF Tensor Working model Every turn
COOP Tensor Cooperative (model signals + Haiku structures) Every turn

This makes the confounds explicit:

  • SELF vs EXT isolates agency (working model vs external) while holding representation and timing constant.
  • BLOCK vs RAW isolates agency (working model vs none) within the conversation representation.
  • EXT vs COMPACT confounds representation AND agency AND timing. Any difference could be any of those three.
  • SELF vs BLOCK confounds representation AND timing. Both have working-model agency, but one uses tensors and the other uses conversation.

Conditions (6)

ID Condition Description
RAW Raw conversation Append-only, no compression. Truncate at window limit. The null baseline.
COMPACT Framework compaction Anthropic's compaction prompt applied to conversation history. What Claude Code does today.
BLOCK Block management March 4 design: model sees block labels, can emit drop/anchor/summarize tags. No tensor. Conversation is the memory, selectively compressed by the model.
EXT External tensor projection Current Hamutay architecture. Haiku projects the tensor from the surface exchange. Model sees tensor as read-only system context.
SELF Self-projected tensor The ALU produces the tensor via tool_use as part of its response. Same schema as EXT. No external projector.
COOP Cooperative tensor The ALU annotates its response with load-bearing signals (what changed, what to keep, what to drop). The Projector uses these annotations as guidance. Hybrid of model judgment and external structuring.

Content Sources (controlled)

All conditions process the SAME content. Two content sources:

  1. Replay: A Pichay conversation log (real tool-use conversation, representative of coding work). The observation_full dataset (104 turns) provides batch-size-controlled content.

  2. Live PRD: The bladnman PRD benchmark. The model plans against a real product requirements document. This tests downstream task quality directly.

Content source and batch boundaries are blocking factors in the analysis, not just operational controls. Each content instance is a unit of replication.

Controls

  • Model: Same model for the ALU across all conditions (Sonnet 4.6). EXT and COOP conditions use Haiku 4.5 for projection.
  • max_tokens: 64000 (streaming) for all calls, per CLAUDE.md.
  • Batch size: For replay, use the same batch boundaries across all conditions within each run. Batch boundaries are a blocking factor.
  • Temperature: 0 where supported, default otherwise.
  • Repetitions: N=5 per condition for replay, N=3 for PRD (cost constrained — PRD runs are more expensive).
  • Context utilization threshold: Defined as (total tokens sent to API) / (model context window size). "High utilization" = > 50%. "Low utilization" = < 25%.

Dependent Variables

Primary: Downstream Task Quality

For PRD content: requirement coverage score (the bladnman evaluation framework — 99 requirements, severity-weighted coverage). This is what actually matters. The model's job is to produce good work, not pretty tensors.

For replay content: not applicable (replay has no downstream task). Replay conditions are evaluated on tensor/memory quality metrics only.

Secondary: Memory System Health (replay conditions)

  1. Riemann dispersion — output coherence clustering (existing metric)
  2. ProcessHealth — metacognitive allocation trend, breathing rhythm, precursor rate, recovery rate (existing metric)
  3. Content preservation — semantic similarity between the information available to the model at cycle N vs the information that existed in cycles 1..N. Measures: does the memory system actually preserve what matters?
  4. Context utilization — what fraction of the context window is occupied by memory overhead (tensor, page table, block labels) vs content that the model can reason about?

Tertiary: Cost

  • Total API tokens consumed per condition (input + output, both models)
  • Cost per cycle at current pricing
  • Cost per unit of downstream quality (cost-effectiveness)

Hypotheses (Pre-Registered)

Primary Contrasts

These three comparisons are the pre-registered tests. Each isolates one design dimension while holding others as constant as possible. Multiple comparison correction (Holm-Bonferroni) applies to these three only.

C1: Agency within tensor representation (SELF vs EXT) Isolates: who curates (working model vs external model). Held constant: representation (tensor), timing (every turn).

  • Prediction: SELF > EXT on PRD coverage by >= 3 percentage points.
  • Mechanism: Self-projection preserves latent intent and task-relevant salience that is not visible in the surface exchange. The external projector operates on an impoverished signal — the text of the exchange rather than the model's internal prioritization of what mattered.
  • Falsification: If EXT >= SELF, external observation is more disciplined than self-assessment. The information asymmetry argument is wrong, or the schema enforcement compensates for the signal loss.

C2: Agency within conversation representation (BLOCK vs RAW) Isolates: model agency (active management vs none). Held constant: representation (conversation), no external model.

  • Prediction: BLOCK > RAW on PRD coverage when context utilization exceeds 50%. No difference at < 25% utilization.
  • Mechanism: Selective compression preserves task-relevant context that naive truncation destroys. The crossover prediction (no difference at low utilization) tests whether the overhead of block management is negligible or costly at small scale.
  • Analysis: Test the condition × utilization interaction term, not just the main effect. A continuous utilization covariate is preferred over binning at 50%.
  • Falsification: If RAW >= BLOCK at all utilization levels, the management overhead (labels, cleanup tags, model attention cost) exceeds the compression benefit.

C3: Representation (SELF vs BLOCK) Isolates: tensor structure vs raw conversation (confounded with timing: every-turn vs on-request). Held constant: agency (working model curates in both).

  • Prediction: SELF > BLOCK on replay ProcessHealth after 50 cycles. BLOCK >= SELF on PRD coverage for short tasks (< 10 cycles).
  • Mechanism: Structured representation with declared losses enables metacognitive recovery after defragmentation events. Unstructured conversation cannot regenerate from breathing. But for short, focused tasks, tensor overhead costs more than it saves.
  • Analysis: Model the condition × cycle interaction to locate the crossover point rather than testing fixed thresholds.
  • Falsification: If BLOCK maintains coherence past 50 cycles, the tensor structure is unnecessary — the model's judgment alone is sufficient. If SELF wins even at cycle 1, the overhead is negligible.

Non-Inferiority Test

C4: Cooperative vs best single-agent condition (COOP vs max(SELF, EXT)) This is NOT a primary contrast because COOP combines dimensions in a way that makes causal attribution difficult. It is pre-registered as a non-inferiority test.

  • Prediction: COOP is non-inferior to the better of SELF and EXT on PRD coverage (margin: -2 percentage points) AND shows better ProcessHealth metrics than the worse of SELF and EXT.
  • Falsification: If COOP is inferior to both SELF and EXT on any primary metric, the coordination overhead hurts more than it helps.

Exploratory Analyses (Not Pre-Registered)

These are examined after the primary contrasts. No multiple-comparison correction. Interpreted as hypothesis-generating, not confirmatory.

  • Does the breathing rhythm differ between SELF and EXT? (Self-projection may change defragmentation — the model that breathes also records its own breathing.)
  • Does BLOCK develop emergent structure over time? (The model may spontaneously organize compressed blocks into something tensor-like.)
  • What is the divergence between SELF and EXT tensors for the same content? (High divergence = the external projector is missing something the model knows.)
  • Do fault handles get used? In BLOCK condition, does the model ever fault back content it previously anchored?
  • How do COMPACT and RAW compare? (Framework compaction is external agency on conversation — the worst of both worlds, or a useful baseline?)
  • Is there a condition × content-type interaction? (Tensor may win for research conversations but lose for PRD planning.)

Protocol

Phase 1: Replay Comparison (all 6 conditions)

  1. Select 50 cycles from observation_full with controlled batch sizes
  2. Run each condition through the same 50 cycles
  3. Collect tensors (or conversation snapshots for non-tensor conditions)
  4. Compute ProcessHealth, Riemann dispersion, content preservation, context utilization for all conditions
  5. N=5 per condition = 30 runs total

Estimated cost: ~$2-4 per run (Haiku projection is cheap; Sonnet self-projection is ~10x more for output tokens). 30 runs ≈ $60-120.

Phase 2: PRD Benchmark (4 conditions)

Run the bladnman PRD evaluation with the projective gateway configured for each condition. Skip RAW and COMPACT — these are the baseline already captured in the March 9 benchmark data.

  1. BLOCK: Gateway provides block labels and processes cleanup tags
  2. EXT: Gateway uses Haiku tensor projection (current architecture)
  3. SELF: Gateway lets the model produce its own tensor
  4. COOP: Gateway provides annotation channel + Haiku structuring

Evaluate with the frozen requirements_catalog_v1 (99 requirements). N=3 per condition = 12 runs total.

Estimated cost: ~$5-15 per run (full PRD processing). 12 runs ≈ $60-180.

Phase 3: Analysis

Pre-specified analysis model:

PRD phase (downstream quality): Paired comparison within each PRD instance. Since all conditions process the same PRD content, the unit of analysis is condition-pair differences in coverage score. Primary test: paired t-test or Wilcoxon signed-rank on the three pre-registered contrasts, with Holm-Bonferroni correction. Report effect sizes (Cohen's d) and 95% CIs for each contrast.

Replay phase (memory health): Mixed-effects model with condition as fixed effect, run replicate as random effect, and cycle as a repeated measure. Content batch boundaries are blocking factors. Primary outcome: ProcessHealth composite at cycles 10, 25, and 50 (pre-specified checkpoints, not chosen after seeing data). For C2 and C3, include a condition × cycle interaction term to test the crossover predictions.

Cost analysis: Report total tokens (input + output, both models) per condition per run. Compute Pareto frontier: downstream quality vs cost per run. A condition that costs 3x more for 1% better coverage is noted but not preferred.

Multiple comparisons: Holm-Bonferroni correction on the three primary contrasts (C1, C2, C3). C4 (non-inferiority) uses a one-sided test with a -2pp margin, analyzed separately. All exploratory analyses reported without correction and flagged as hypothesis-generating.

Reporting: All data, including individual run scores and tensor logs, published alongside the findings document. Failed runs reported, not excluded.

Total estimated cost: $120-300

Implementation Requirements

New code needed:

  1. Self-projection mode for ChatSession — ALU produces tensor via tool_use alongside its response. Single API call.

  2. Cooperative mode — ALU emits lightweight annotations (what changed, what's load-bearing, what to drop). Projector reads annotations. Two API calls but the Projector is informed.

  3. Block management mode — Implements March 4 design: block labeling, cleanup tag processing, KV store for anchored/summarized blocks, fault handle resolution. No tensor.

  4. Experiment harness — Runs same content through all conditions, collects outputs, computes metrics. Extends existing ablation harness pattern.

  5. Content preservation metric — New eval metric: semantic similarity between what the model can access at cycle N vs what existed in cycles 1..N. Uses existing BGE-large embeddings.

Existing infrastructure reused:

  • Projector class (EXT condition, COOP condition base)
  • TensorLog (all tensor conditions)
  • ProcessHealth, Riemann dispersion (all conditions, adapted for non-tensor conditions where applicable)
  • Bladnman evaluation framework (Phase 2)
  • observation_full dataset (Phase 1)

Decision Criteria

After data collection, the architecture decision follows:

  • If one condition dominates on both downstream quality and tensor health, adopt it.
  • If there's a quality/cost tradeoff, present the Pareto frontier and let the data inform the choice.
  • If the tensor conditions don't beat BLOCK on downstream quality, the tensor representation needs to justify its existence on other grounds (identity transfer, cross-instance sharing, Yanantin integration) or be reconsidered.
  • If SELF and EXT produce similar quality but SELF costs significantly more, the external projector earns its keep despite the architectural concerns.

What This Experiment Does NOT Test

  • Multi-model tensor sharing (Yanantin integration)
  • Identity transfer across instances
  • Sessions longer than 104 cycles
  • Content types other than coding conversation and PRD planning
  • Edge-enhanced tensors (the depends_on/shed_from schema extension)
  • The effect of model size on any of these strategies

These are important questions but require separate experiments.
